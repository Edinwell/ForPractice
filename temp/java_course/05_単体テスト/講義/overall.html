テスト概論

テストへの道

ソフトウェア開発全体の中で、ソフトウェアのテストというものはとても大きな割合を占めており、非常に重要な工程です。

また、それらはテストをする範囲や対象、順序によって分類されています。
本研修で学ぶ単体テストとは、その中で一番初めに行うテストです。

この章ではまず、ソフトウェアのテストの大まかな流れと、
その中で単体テストがどういう位置づけにあるかを学んでいきましょう。

なぜテストが必要なのか？

一般紙においても「IT」という単語が現れない日はないと言えるほど、今日この技術は社会に広く浸透しています。
インターネット、そしてさまざまなソフトウェア技術の加速度的な発展とともに、現在では、数えきれないほどのプロジェクトにより
開発作業が行われています。

しかし、いくら技術が発展しても、開発の手戻りや致命的なバグの発生といった問題は
今なおソフトウェア開発の最重要課題として、多くのプロジェクトのメンバーを苦しめています。

この問題を解決するための工程が、ソフトウェアのテストです。

プログラムの欠陥を修正する時のコストは、発見が遅れれば遅れるほど雪だるま式に増えていきます。
そのため、作業工程の節目節目でしっかりとテストをし、その時点での品質を保証することで、
ある工程で発見するべき欠陥を見過ごして後の工程に影響を及ぼすという事態を防ぎ、
欠陥修正のコストを軽減するのです。

では、ソフトウェアのテストとは具体的にどのような流れになるかを、これから見ていきましょう。

ソフトウェアテストの流れ

一般的なソフトウェアテストは下図のような流れとなります。


ソフトウェアの開発プロジェクトにおいては、「開発コストは欠陥除去コストに左右される」 といっても 過言ではなく、
それぞれのフェーズで、下記のような事を確認します。

単体テスト

  • モジュール単体の機能確認
  • 内部ロジックの確認
  • 条件分岐を網羅していることの確認
  • 例外条件やその処理方法の確認

結合テスト
  • サブシステム内のモジュール間の確認
  • サブシステム間のインタフェース部分の動作を確認

システムテスト
  • 新機能テスト（正常／異常／復旧［New Function Testing］）
  • 回帰テスト（Regression Testing）
  • 負荷テスト（Load/Stress Testing）
  • 環境変化に対するテスト（Environment Testing）
  • パフォーマンス測定テスト（Performance Measrement Testing）

受け入れテスト（運用テスト）
  • 実際の稼働環境下において実施の確認
  • シナリオによる確認
